# Лабораторная работа №1. Последовательное умножение квадратных матриц

## 1. Описание алгоритма

### 1.1 Порядок обхода i-k-j

Стандартный алгоритм умножения матриц C = A × B:

```
C[i][j] = Σ(k=0..N-1) A[i][k] * B[k][j]
```

В реализации используется порядок циклов **i-k-j**:

```cpp
for (int i = 0; i < n; i++) {
    for (int k = 0; k < n; k++) {
        double a_ik = A[i * n + k]; // загружаем один раз
        for (int j = 0; j < n; j++) {
            C[i * n + j] += a_ik * B[k * n + j];
        }
    }
}
```

**Особенности i-k-j:**
- Внутренний цикл по `j` обращается к `B[k][j]` и `C[i][j]` последовательно по строке (stride=1)
- Значение `A[i][k]` вынесено во временную переменную — исключается повторное обращение к памяти
- Этот порядок более дружественный к кэшу, чем стандартный i-j-k, **но при больших N матрицы перестают помещаться в кэш**, что снижает производительность

### 1.2 Хранение матриц

Матрицы хранятся в одномерных массивах (`new double[n*n]`) с линейной индексацией `[i*n+j]`.

### 1.3 Вычислительная сложность

- Количество операций: **2N³** (N³ умножений + N³ сложений)
- Сложность по памяти: **3N²** элементов типа `double` (24N² байт)

## 3. Верификация

Результат умножения проверяется скриптом `verify.py`:
1. Загружает матрицы A, B и результат C++ из файлов
2. Вычисляет эталонное произведение `C_ref = A @ B` через NumPy (BLAS)
3. Сравнивает по относительной ошибке нормы Фробениуса

Критерий: относительная ошибка < 10⁻⁶.

## 5. Анализ результатов

### 5.1 Временная сложность
Экспериментальные данные подтверждают теоретическую сложность **O(N³)**:
- При увеличении размера матрицы N от 200 до 2000 (в 10 раз) время выполнения выросло с 0.003 сек до 4.451 сек (примерно в 1483 раза, что близко к теоретическому $10^3 = 1000$ с поправкой на накладные расходы при работе с памятью).
- На совмещенном графике кривая фактического времени практически ложится на кривую теоретической сложности O(N³).

### 5.2 Производительность и влияние кэша
Производительность в GFLOP/s **не является стабильной** — она демонстрирует характерный профиль с ранним пиком и последующим падением:

| Диапазон N  | Производительность (GFLOP/s) | Причина                                     |
|-------------|----------------------------|----------------------------------------------|
| 200 → 400   | 4.94 → **6.67** (Пик) ↑     | Амортизация накладных расходов, матрицы полностью помещаются в быстрый кэш |
| 400 → 2000  | **6.67** → 3.59 ↓           | Постепенное вытеснение данных из кэша L3, рост числа обращений к медленной ОЗУ |

**Объяснение пика при N=400:**
- При **N=200** время выполнения слишком мало (0.003 с), из-за чего накладные расходы на вызов функций, организацию циклов и управление памятью "съедают" часть производительности.
- При **N=400** достигается "золотая середина". Три матрицы суммарно занимают всего **~3.7 МБ**. Этот объем целиком и с запасом помещается в кэш-память L3 (а частично и в L2) современных процессоров, обеспечивая максимально быстрый доступ к данным без задержек.
- При **N=800** и выше рабочий набор данных вырастает до **14.7 МБ** и более. Это начинает превышать объем L3 кэша типичного процессора. Возникают промахи кэша (cache misses), процессору приходится простаивать в ожидании данных из оперативной памяти, из-за чего производительность неуклонно падает вплоть до 3.59 GFLOP/s при N=2000 (размер данных 91.5 МБ).

### 5.3 Память
Потребление памяти растёт как **O(N²)** и строго определяется формулой:
`Память = 3 × N² × 8 байт`
- **N=200:** ~0.9 МБ - легко помещается в кэш любого уровня.
- **N=400:** ~3.7 МБ - оптимально для L3 кэша.
- **N=2000:** ~91.5 МБ - значительно превышает размер кэша процессора, полностью располагаясь в оперативной памяти.

## 6. Выводы

1. **Реализован** базовый алгоритм умножения квадратных матриц с оптимизированным порядком обхода циклов `i-k-j` (по строкам, что улучшает пространственную локальность данных). Реализация - однопоточная, без использования сторонних математических библиотек.

2. **Экспериментально подтверждена** кубическая теоретическая сложность алгоритма - O(N³). Рост затрат времени следует графику теоретической функции.

3. **Обнаружена и обоснована выраженная зависимость производительности от архитектуры кэш-памяти:**
   - Максимальная эффективность вычислений (**6.67 GFLOP/s**) зафиксирована при N=400. На этом этапе объем обрабатываемых данных (~3.7 МБ) идеально укладывается в кэш-память процессора.
   - По мере роста матриц до N=2000 производительность деградирует до **3.59 GFLOP/s** (падение на ~46%), так как объем данных (91.5 МБ) требует постоянного взаимодействия с более медленной оперативной памятью (bottleneck пропускной способности памяти).
